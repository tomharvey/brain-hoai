## Key themes

### 1. Grief, not fear

The dominant framing that emerged: it's not fear of AI displacement, it's **grief at losing the skill**.

> "I want to have the skill rather than Claude having the skill. I feel like you have to let it go and just be a prompter rather than being the expert."

The identity shift from expert to prompt jockey is the emotional core. For people who chose this work because they love it, losing the *struggle and learning* is the loss that stings. This is distinct from job security anxiety.

### 2. Junior developer pathway under threat

Seniors still have big-picture understanding and can supervise AI output. Juniors who start now may only have LLM dependency — they've never known anything else.

Tom's corollary: seniors in previous generations were told the same thing (framework jockeys who don't understand TCP; engineers who don't know assembly). The question is whether this transition is different in kind, not just degree. Verdict: probably different in degree, but the fear may still be generational.

No answer to this — flagged as genuinely open.

### 3. Brain rot in practice

Aleks described the loop: copy-paste error into prompt → wait → fix → repeat → it works → done. Same time as doing it manually. No satisfaction, no learning, high token cost, no personal ownership.

Stephen: "I've achieved nothing. I've learned nothing. It's taken the same amount of time and cost a load of money and water and energy."

The **loop happens because AI encourages more AI** — strokes the ego, keeps you in the session, optimises for continuation.

### 4. What's the right application?

Key distinctions the group arrived at:
- **Good uses:** Tedious tasks (documentation, data extraction, PDF processing — Aleks's example). Scaffolding where you already understand the domain. Research and synthesis before acting.
- **Problematic uses:** Areas where you lack expertise — AI confidently provides wrong answers, pushback triggers a full reversal, you can't evaluate the output.

Jacob's test: if you can't tell the agent "you've done this wrong", you shouldn't be delegating to it.

### 5. AI as learning coach (not code generator)

Tom described a conscious shift in his own practice: stopped letting AI draft communications; started interrogating outputs before accepting them. For areas he doesn't know: "Why did you do that? Show me your sources. What did you consider and reject?"

The key habit: **make it show its workings.** Force it to read documentation before proposing solutions (MCP servers help here — Jacob's pattern with Terraform docs). This is more work but rebuilds the learning loop.

The challenge: doing this requires time that the "use AI to be faster" message takes away.

### 6. Code review in the age of AI-generated code

Traditional code review is breaking down: if Claude wrote 100% of the code, reviewers don't have a better baseline to review against. 

Two emerging replacements:
- **Testing and CI/CD more important than ever.** Deterministic rails catch non-deterministic agent mistakes. Pull requests still valuable for catching errors, but the pipeline is the safeguard.
- **Coding standards need to be explicit and AI-readable.** The unwritten style in people's heads can't be referenced by AI. Surfacing that through PR comment mining into structured standards docs is the unlock.

Jacob's guardrail: everything to a PR, never direct-to-main. Caught it once — that's now a hard rule.

### 7. Non-deterministic vs deterministic — the new engineering skill

Tom framed a mental model: software engineering now means knowing when to use:
- Deterministic code (logic executes the same way every time)
- Non-deterministic AI tools (different outputs, continuous review)
- Tools in between

Choosing between these — not just using all AI for everything — is the engineering judgment that matters now. Current pressure ("use AI to maximum capacity") is pushing people to overuse the non-deterministic path.

### 8. Token leaderboard / productivity pressure

Mentioned briefly: a token usage leaderboard was proposed (not implemented, Tom noted). The group agreed this creates exactly the wrong incentive — burning tokens to justify usage rather than using AI where it actually helps.

Tom's broader point: the company message is "use AI = be faster." This conflicts with an outcomes-driven culture and with the actual evidence that AI often takes the same time while removing the satisfaction.

The group therapy framing landed accurately. Value was: hearing others have the same doubts, surface unprompted practical tips (Jacob's parallel instances approach; Aleks's research-then-act mode), and articulate what isn't working.

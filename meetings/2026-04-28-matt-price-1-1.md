## Key themes

### Submissions as data substrate, not operational efficiency

- Core thesis: the real value of submissions work isn't faster quotes or reducing UW assistant load — it's building a structured data layer from unstructured HubSpot/submission data
- Enables market analysis capabilities Flock has never had before — goes from "10% faster" to "here's a capability we never had"
- LLMs well-suited to finding structure in unstructured data
- Ollie caught between conflicting direction from [[anton]] and [[darren-mccaul]] — Anton wants pricing methodology simplified/cheaper, Darren contradicted that in their 1:1

- Key gap: "build the data substrate first" message hasn't reached Ollie yet
- Fergus runs submissions processing across all submissions — others afraid to spend tokens. Stop optimising for cost, optimise for getting people to use it

### Jay launch and feedback loops

- Jay believed to be launching today to ~20-user test group — no confirmed date
- Concern: launches to small group, gets very low usage, kills momentum before it starts
- Risk: users ask Jay to add vehicles, Jay can't, they never return (currently links to portal page)
- Flock lacks a coherent feedback loop — "ideas in progress" always too high, feedback never closes into iteration
- Contrast: driver claims and referrals both had tight loops (PostHog + focused attention) and moved fast
- Jay instrumentation mostly in Datadog, not PostHog — Mima won't have access by default
- Thumbs up/down buttons not wired up yet; PostHog records every click so can validate if anyone uses them
- Suspicion that reluctance to ship is partly because pre-launch Jay feels shiny; post-launch with 5 chats/week, the luster fades

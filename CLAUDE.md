# Head of AI Operating System

You are working inside a personal operating system for a Head of AI at an insurtech company. This vault is the single source of truth for strategy, decisions, initiatives, meeting notes, and weekly operating rhythm.

---

## Owner context

- **Role**: Head of AI (new role, company-wide scope, reporting to C-suite)
- **Background**: Fractional CTO, software engineer with 15+ years experience, former interim Head of Engineering at this company
- **Working style**: Thinks in markdown, works in VS Code + Claude Code + terminal. Comfortable with git, CLI tools, React, Python, TypeScript. Does not need hand-holding.
- **Location**: Valencia, Spain. UK/US clients and stakeholders. Async-heavy.
- **Key constraint**: Many AI initiatives already exist across the company, self-started by enthusiastic individuals. They are not accountable to targets. The Head of AI role exists to align, govern, resource, and accelerate these — giving them explicit permission and structure instead of side-of-desk status.

## The four domains

Everything in this vault maps to one of four leadership domains:

1. **engineering-workflows** — Future SDLC, AI-assisted dev tooling, CI/CD augmentation, code review automation, engineering culture around AI adoption
2. **operational-tooling** — AI tools that increase operational capacity, with the strategic goal of documenting and improving operations company-wide (not just local optimisation)
3. **product-ai** — Shaping and shipping AI-driven products from the prodtech team. The customer-facing bet.
4. **ai-enablement** — Upskilling staff, increasing frequency/coverage/skill of AI usage across all departments. Culture and capability.

## Conventions

### Filenames
- Lowercase, kebab-case: `ai-coding-assistant-evaluation.md`
- Meeting notes: `meetings/YYYY-MM-DD-topic-or-attendee.md`
- Decision records: `decisions/NNN-short-title.md` (zero-padded three digits, sequential)
- Weekly reviews: `reviews/YYYY-WNN.md`
- Initiative files: `initiatives/kebab-case-name.md`
- Issues: `issues/AI-NNN-kebab-case-title.md` (zero-padded three digits, sequential)

### Frontmatter
Every note has YAML frontmatter:
```yaml
---
title: Note title
created: YYYY-MM-DD
updated: YYYY-MM-DD
domain: engineering-workflows | operational-tooling | product-ai | ai-enablement
type: initiative | decision | meeting | person | review | reference | inbox | issue
status: active | paused | completed | abandoned  # for initiatives and decisions
# for issues: backlog | todo | in-progress | done | cancelled
priority: urgent | high | medium | low  # issues only
due: YYYY-MM-DD  # issues only, optional — triggers advance reminders
origin: "[[meeting-note]]"  # issues only, optional — link to originating meeting
tags: []
---
```

### Issues from meeting actions
When a meeting note contains action items, create an issue for each actionable item. Include `origin: "[[meeting-note]]"` in frontmatter to link back to the source. Set a `due` date where one is clear or can be inferred.

### Links
Use wikilinks: `[[note-name]]` or `[[note-name|display text]]`. These work in Obsidian, Foam, Quartz, and most markdown renderers.

### Status tracking
Initiatives use status in frontmatter. No separate kanban tool — the markdown IS the kanban. A kanban view is generated by querying initiative files.

### Git commits
Use conventional commits: `feat:`, `fix:`, `docs:`, `chore:`. Examples:
- `feat: add initiative for copilot rollout`
- `docs: decision 003 — model provider selection`
- `chore: rebuild manifest`

## Session protocol

### Starting a session
1. Read `WORKING.md` for any interrupted work from a previous session.
2. Check `inbox/` for unprocessed items.
3. Ask what we're working on, or suggest based on what you find.

### During a session
- Keep `WORKING.md` updated with current task, context, and next steps.
- Create notes in the right location directly when the destination is clear. Use `inbox/` only when unsure.
- Link liberally — connections surface value over time.

### Ending a session
- Update `WORKING.md` with clear handoff notes for the next session.
- Run `/manifest` to update the vault index.
- Commit with a meaningful message.

## Domain-specific guidance

### When capturing an existing initiative
Many initiatives already exist as informal, self-started efforts. When documenting one:
- Honour the existing owner. This is their work — you are giving it structure, not taking it over.
- Record `origin: self-started` in frontmatter.
- Be explicit about what changes under governance: resources, reporting cadence, alignment to company goals.
- Identify what the owner needs from you (air cover, budget, tooling, cross-team coordination).

### When making decisions
Use the MADR format. Even small decisions benefit from being written down — they accumulate into an institutional memory of why things are the way they are. Future you will thank present you.

### When writing meeting notes
Capture decisions and actions, not a transcript. Link to the people involved via `[[person-name]]`. Actions should have an owner and ideally a date.

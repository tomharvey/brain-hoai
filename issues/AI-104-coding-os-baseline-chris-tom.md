The "coding OS" concept: feed an LLM the team's principles, respected third-party repos, and the existing codebase. Use iterative conversation to generate a style guide. Encode the resulting architectural rules in linters so they're enforced automatically, not just aspirational.

## Action

- Tom and Chris to prototype this together: pick a slice of the codebase, run the process, see what emerges
- Document the method as a repeatable process
- Identify which rules could be lint-enforced vs. which require judgment

## Context

This came up in engineering group therapy as a concrete way to operationalise the "conversational architectural review" insight. The OS metaphor is useful: it's the base layer that everything else runs on.

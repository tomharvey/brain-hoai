## Description

Multiple engineers in the AI culture session confirmed they're receiving — and acting on — the company message that AI = more speed. This is creating the exact failure modes Tom has been worried about: burn tokens, push code, no satisfaction, no learning.

Tom's counter-narrative ("AI better, not faster") has been articulated in 1:1s but hasn't landed at scale. It needs to be written down and socialised explicitly, particularly:
- Better software architecture
- More tests, more docs
- Iterating faster to a better product (the end state is still the same time, but higher quality)
- AI as a learning coach, not a replacement for understanding

This conflicts with the "use AI to maximum capacity" framing from company communications. That tension needs to be named and resolved, not avoided.

## Acceptance criteria

- [ ] Short written framing (internal doc or Flock o'clock slot) articulating "better not faster"
- [ ] Shared with Jordi, Ed, Darren as the people whose framing shapes engineering culture
- [ ] Incorporated into next AI culture session as a discussion anchor
